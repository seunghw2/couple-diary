# 07. 심사 거절 흔한 사유 체크리스트

App Store Review Guidelines에서 자주 거절되는 항목을, **우리 앱 '투데이'가 해당되는지 + 해야 할 조치**와 함께 정리했습니다. 각 항목의 상태 표기:

- 🔴 **해당(심사 blocker)** — 지금 제출하면 거절 위험 큼
- 🟡 **부분 해당** — 조건부/보완 필요
- ✅ **양호** — 현재 문제 없음

---

## 4.8 — Sign in with Apple (제3자 로그인 앱의 핵심)

**상태: 🔴 해당 — 반드시 다뤄야 함**

Apple 가이드라인 4.8: 앱이 **제3자/소셜 로그인(예: 카카오, 구글, 페이스북)**으로 사용자의 **주 계정을 만들거나 인증**하면, 다음 세 조건을 모두 만족하는 **동등한 로그인 옵션을 하나 더** 제공해야 합니다.

1. 수집 데이터를 사용자 이름과 이메일로 제한
2. 이메일 비공개(가리기)를 선택할 수 있음
3. 광고 목적의 앱 사용 데이터 수집을 동의 없이 하지 않음

**우리 앱 현황:** 로그인은 **카카오 단독**(`frontend/app/(auth)/login.tsx`). 그 외 실사용 로그인은 없음(닉네임 로그인은 dev용). 카카오는 위 3조건을 만족하는 "동등 옵션"으로 인정되지 않으며, 4.8 예외(자체 계정 시스템만 사용 / 교육·기업 계정 / 정부·산업 ID / 특정 서드파티 서비스 클라이언트)에도 해당하지 않습니다.

→ 따라서 **"카카오만 유지"는 불가**합니다. 아래 중 하나가 필요:

### 옵션 A — Sign in with Apple 추가 (권장, 가장 확실)

동등 옵션 3조건을 태생적으로 만족합니다. iOS에서 가장 마찰 없이 통과.

구현 개요:
1. **의존성**: `npx expo install expo-apple-authentication` 설치.
2. **엔타이틀먼트**: app.json에 `ios.usesAppleSignIn: true` 추가(관리형에서 자동으로 `Sign in with Apple` capability를 붙임). Apple Developer의 App ID에도 해당 capability가 켜져 있어야 함(EAS 자동관리 시 처리됨).
3. **프론트**: 로그인 화면에 Apple 버튼 추가 → `AppleAuthentication.signInAsync()`로 `identityToken`(+최초 1회 name/email) 획득.
4. **백엔드 토큰 검증**: 프론트가 받은 `identityToken`(JWT)을 백엔드로 전송 → 백엔드가 Apple 공개키(https://appleid.apple.com/auth/keys)로 서명 검증하고 `sub`(Apple 사용자 식별자)를 기존 카카오 사용자와 같은 방식으로 매핑해 우리 JWT 발급. (현 구조상 `backend/.../auth/`에 `AuthController` + `KakaoClient`가 있으므로 같은 패턴으로 `AppleClient`/엔드포인트 추가.)
5. **계정 삭제 연동**: Apple은 계정 삭제 시 **Sign in with Apple REST API로 토큰 revoke**를 요구(5.1.1(v)와 연결). 계정 삭제 서비스에서 revoke 호출을 함께 처리.
6. **주의**: 최초 로그인에서만 name/email이 오므로 백엔드가 그때 저장해야 함. 재로그인 때는 `sub`만 옴.

### 옵션 B — 자체 이메일/비밀번호 로그인 추가

3조건(이름·이메일만 수집, 이메일 비공개는 자체 시스템이라 무관, 광고 추적 없음)을 만족하는 자체 계정 시스템도 동등 옵션으로 인정됩니다. 다만 iOS에서는 Apple 로그인이 사용자 경험·통과율 면에서 더 안전하므로 **옵션 A를 우선 권장**합니다.

> 참고: 2024년 이후 Apple은 문구를 다소 완화했지만, "카카오 같은 소셜 로그인만 제공 + 위 3조건 미충족"인 우리 케이스는 여전히 동등 옵션 제공 대상입니다.

## 5.1.1(v) — 인앱 계정 삭제

**상태: 🟡 부분 해당 — 백엔드는 준비, 프론트 미구현**

계정 생성을 지원하는 앱은 **앱 안에서 계정 삭제**를 제공해야 합니다. 삭제를 불필요하게 어렵게 만들면 거절됩니다.

**우리 앱 현황:**
- 백엔드에 `DELETE /api/me`가 이미 있고(`backend/.../user/UserController.deleteAccount`, 주석에 "Apple 5.1.1(v) 필수"까지 명시), 커플 공유 데이터까지 하드 삭제하도록 구현됨.
- 그러나 **프론트에 이를 호출하는 UI가 없음**. `app/account.tsx`에는 로그아웃만 있고 삭제 없음. `lib/api.ts`의 `authApi`에도 삭제 메서드 없음.

→ 조치:
1. `authApi`에 `deleteMe: () => api.del('/api/me')` 추가.
2. `app/account.tsx` 하단에 "회원 탈퇴" 진입점 추가 → 2단계 확인(`confirmAsync`) → 삭제 → 토큰 정리 후 로그인 화면으로.
3. 카피에 "탈퇴 시 내 일기와 커플 공유 데이터가 영구 삭제되며 복구할 수 없다"를 명시.
4. Sign in with Apple을 붙였다면 삭제 시 Apple REST API로 token revoke도 수행.

## 5.1.1 — 권한 목적 문자열 (Purpose Strings)

**상태: ✅ 대체로 양호 — 위치는 검토**

각 권한 요청에 **왜 필요한지** 명확한 한국어 설명이 있어야 하고, 실제 기능과 일치해야 합니다.

**우리 앱 현황(`app.json`):**
- `NSPhotoLibraryUsageDescription`: "일기에 사진을 첨부하기 위해…" — 사진 첨부 기능과 일치 ✅
- `NSPhotoLibraryAddUsageDescription`: 사진 저장용 ✅
- `NSLocationWhenInUseUsageDescription`: "일기에 장소를 기록할 때…" — 장소/지도 기능과 일치 ✅
- `expo-media-library` 플러그인의 photosPermission 문구도 존재 ✅

→ 조치: 실제로 위치 권한을 **런타임에 요청하는 코드가 있는지** 확인. 문구만 있고 권한을 안 쓰거나, 반대로 쓰는데 문구가 빠지면 문제. 카메라를 쓰면 `NSCameraUsageDescription`도 필요(현재 image-picker가 카메라 촬영을 지원하면 추가).

## 2.1 — 앱 완성도 / 크래시 · 버그

**상태: 🟡 부분 해당**

리뷰 중 크래시나 명백한 버그가 있으면 거절됩니다.

**우리 앱 현황:** 로딩/에러/빈 상태 처리가 전반적으로 좋음(감사 문서 참고). 다만:
- **터널 백엔드(hammerslog.trade)**가 심사 중 죽으면 앱 전체가 로그인 불가 → 사실상 크래시급 거절. 프로덕션 서버 필수(5.6/2.1, `09` 참고).
- 오프라인/서버 다운 시 사용자에게 명확한 안내가 나오는지 확인.

---

## 4.2 — 최소 기능 (Minimum Functionality)

**상태: ✅ 양호**

단순 웹 래퍼나 기능이 빈약하면 거절됩니다. 이 앱은 일기 작성/조회, 커플 연결, 오늘의 질문, 지도·장소, 알림 등 충분한 네이티브 기능을 가집니다. ✅

→ 조치: **"닉네임으로 시작 (개발용)"** 로그인이 프로덕션 화면에 그대로 보이면 "미완성/테스트 기능" 인상을 줘 감점 요인. 프로덕션 빌드에서 숨기거나 데모 전용으로 전환(`app/(auth)/login.tsx`).

---

## 심사 중 백엔드 가용성

**상태: 🔴 해당**

로그인·데이터가 서버에 의존하는 앱은 **심사 기간(및 그 이후) 백엔드가 24/7 정상**이어야 합니다. 개발용 터널(cloudflared/ngrok, `hammerslog.trade`)은 세션이 끊기거나 재시작되면 다운 → 리뷰어가 아무것도 못 봐 즉시 거절.

→ 조치: 고정 도메인 + HTTPS의 상시 가동 프로덕션 서버로 이전(`09-production-backend.md`). `eas.json`/app.json의 API URL을 실주소로.

---

## 데모 계정 제공 (로그인 필요 앱)

**상태: 🔴 해당**

로그인 없이는 기능을 못 보는 앱은 **작동하는 데모 계정**을 App Review Information에 제공해야 합니다. 리뷰어가 카카오 로그인을 실제로 통과하기 어려울 수 있으므로:

→ 조치:
- 미리 만들어 둔 **테스트 카카오 계정 자격증명**을 리뷰 노트에 기재, 또는
- 심사용으로만 열어 두는 안정적인 로그인 경로(예: 데모 계정) 제공.
- **커플 연결**이 필요한 기능은 재현이 까다로우므로, 이미 커플로 연결되고 샘플 일기가 있는 데모 계정을 준비하고 접속 절차를 상세히 적기.

## 5.1.2 / App Privacy — 개인정보 라벨 정확성

**상태: 🟡 부분 해당**

App Store Connect의 App Privacy(영양성분표) 신고가 실제 수집과 일치해야 합니다.

**우리 앱이 수집하는 것(코드 기준):** 이메일·닉네임(카카오), 사용자 콘텐츠(일기 본문·댓글), 사진(첨부), 위치(장소 기록), 생일/기념일.

→ 조치: 위 항목을 빠짐없이 정확히 신고. 실제로 안 쓰는 항목을 신고하거나, 쓰는데 빠뜨리면 거절. **개인정보 처리방침 URL도 필수**(현재 앱/스토어 모두 없음 → `08` P0).

---

## 2.3 / 메타데이터 · 스크린샷 일치

**상태: 🟡 준비 필요**

스크린샷·설명이 실제 앱 화면·기능과 달라선 안 됩니다.

→ 조치: 실제 화면으로 캡처(합성/과장 금지), 설명에 없는 기능을 스크린샷에 넣지 않기, 반대로 스크린샷에 있는 화면은 앱에 실제 존재해야 함.

## placeholder / lorem ipsum 금지

**상태: 🟡 확인 필요**

임시 텍스트·더미 데이터·"준비 중" 화면이 남아 있으면 거절.

→ 조치:
- `eas.json`의 `YOUR_APPLE_ID_EMAIL`, `api.example.com` 등 placeholder는 빌드 전 제거(런북 2.2).
- 앱 내 빈 화면/미완성 문구, 지도 "키 필요" 안내 등이 프로덕션에서 노출되지 않는지 확인.
- 스토어 설명/키워드에 lorem·테스트 문구 없기.
