# 15 · 장소 추가 방식 3안 목업 비교

> **날짜**: 2026-07-05
> **배경**: 현재 일기 작성 화면의 "다녀온 장소" 추가는 **카카오 검색 시트(텍스트 리스트)** 한 가지뿐.
> 검색 → 결과 한 줄씩 → 하나씩 탭이라 흐름이 단조롭고, 자주 가는 단골·최근 방문을 매번 다시 쳐야 한다.
> 서브에이전트 3명이 각기 다른 방향으로 장소 추가 UX를 다시 그렸다.

각 안은 **① 추가 화면 · ② 선택된 상태 · ③ 일기에 담긴 결과** 3프레임 스토리보드.

---

## 한눈 비교

| | 컨셉 | 주 동선 | 강점 | 약점 |
|---|---|---|---|---|
| **A안 · 지도에서 콕** | 지도 우선 | 지도 둘러보기 → 핀 탭 → 카드 → 추가 | 위치감·탐색 재미, "어디였지" 회상에 강함 | 카카오 WebView 마커 상호작용 구현 무겁고, 정확 검색엔 느림 |
| **B안 · 원탭 추천** | 추천 칩 | 최근·단골·근처 칩을 **탭 한 번** | 반복 방문(단골)에 압도적으로 빠름, 구현 가벼움 | 처음 가는 곳은 결국 검색 필요 |
| **C안 · 리치 검색 멀티선택** | 검색 강화 | 검색 → 카드 여러 개 **체크 → 한 번에 추가** | 하루 여러 곳 다닌 날 효율적, 카테고리·거리 정보 풍부 | 단골 반복엔 매번 검색이라 B보다 손이 감 |

## A안 — 지도에서 콕

![A안](assets/15-place-add/mockups/a-mappick/shot.png)

검색 텍스트 리스트 대신 **지도를 직접 보며 핀으로 추가**. 최근·근처 후보가 코럴 핀으로 지도에 흩뿌려지고, 하단 시트엔 "이 근처 우리가 좋아할 만한 곳". 핀을 탭하면 장소 카드(카테고리·별점·주소)가 뜨고 "이 장소 추가". 일기엔 "지도에서 콕 찍은 그곳"으로 담긴다.

- 👍 위치 기반 회상("한강 근처 어디였는데")에 가장 자연스럽다. 지도 자산을 이미 쓰는 앱 성격과 맞음.
- 👎 react-native-webview 안에서 마커 탭 → RN 브릿지로 카드 띄우기가 무겁다. 상호명 정확 검색은 지도 방식이 오히려 느림.

## B안 — 원탭 추천

![B안](assets/15-place-add/mockups/b-quickchips/shot.png)

검색을 뒤로 밀고 **추천 칩 탭 한 번**이 주인공. `최근 간 곳`(지난 일기에서) · `자주 가는 곳`(우리 단골, 방문 N번) · `지금 근처`(현재 위치) 세 묶음. 칩을 탭하면 코럴+체크로 바로 담기고, 하단 "3곳 추가하기". 직접 찾을 때만 상단 검색.

- 👍 커플은 대개 **가던 곳을 또 간다** — 단골/최근을 탭 한 번은 실사용에서 가장 빠르다. 방문횟수 데이터(이미 map 탭에서 집계)를 그대로 재활용, 구현 가벼움.
- 👎 처음 가는 곳은 결국 검색으로 빠져야 함(그래서 검색은 유지).

## C안 — 리치 검색 멀티선택

![C안](assets/15-place-add/mockups/c-richsearch/shot.png)

지금의 검색 시트를 **카드형으로 리치하게 + 멀티선택**. 결과가 카테고리 색 태그·거리(320m)와 함께 카드로 뜨고, 여러 곳을 체크한 뒤 "3곳 한 번에 추가". 일기엔 각 장소가 X(삭제) 달린 칩으로 담긴다.

- 👍 하루에 카페→식당→공원 여러 곳 다닌 날, 한 번 검색해 **한꺼번에** 담기 편하다. 기존 `placeApi` 검색을 그대로 확장 — 구현 부담 적음.
- 👎 단골 반복엔 매번 같은 걸 검색해야 해서 B보다 손이 더 간다.

---

## 추천 — B + C 하이브리드

세 안은 배타적이지 않다. 실제 사용 패턴은 두 갈래다:

1. **또 가는 단골/최근** → B안의 원탭 추천이 최적
2. **처음 가는 곳 / 하루 여러 곳** → C안의 리치 검색 + 멀티선택이 최적

그래서 **기본 화면은 B안(추천 칩 원탭)** 으로 열고, 상단 검색을 누르면 **C안(카드 멀티선택)** 으로 전환하는 하나의 시트를 제안한다.
A안 지도 방식은 매력적이지만 WebView 마커 상호작용 비용이 커서, **map 탭의 별도 탐색 경험**으로 두고 작성 플로우엔 넣지 않는다.

- 구현 재활용: 방문횟수·최근 장소 집계(map 탭), `placeApi.search`(현 검색 시트) 둘 다 이미 있음 → 하이브리드는 대부분 조립.
- 단계: (1) 현 검색 시트를 카드형 멀티선택으로 리치화(C) → (2) 그 위에 추천 칩 섹션 얹기(B).

> 결정 대기: 하이브리드로 갈지, 단일 안으로 갈지. 확정되면 KakaoPlaceSearch 시트를 이 방향으로 개편한다.
