# 16 · 지도 장소 선택 기능 — 갭 분석 & 기획

> **날짜**: 2026-07-05
> **배경**: [15a](15a-place-add-mappick.md)로 "지도에서 장소 콕 찍어 담기"를 구현했지만, 실사용에서 **일반 지도앱이 당연히 주는 기본기**가 빠져 이질감·오선택이 생긴다는 피드백. 사용자가 짚은 두 가지 구체 문제 + 경쟁 앱 대비 결손을 정리하고 우선순위를 제안한다.

---

## 0. 현재 구현 요약 (기준선)
`components/KakaoMapPicker.tsx` 기준 이미 있는 것:
- 상단 검색바(디바운스 350ms) → 백엔드 프록시(`/api/places`) 키워드 검색 → 결과 핀 + 카메라 이동
- 지도 롱프레스 → 좌표 핀 + 역지오코딩 주소 + 이름 입력 → 좌표까지 담기
- 핀 탭 → 장소 카드(이름/카테고리/거리/주소) → 멀티선택 바스켓, 담김✓ 토글
- 검색 0건 폴백(이름만 등록 / 지도에서 직접), 네트워크 오류 재시도
- 현재위치 중심 열림 + 파란 내 위치 점(권한 거부 시 서울시청 폴백), 거리(하버사인) 표시

**핵심 제약**: 탭 가능한 핀은 **검색핀·롱프레스핀뿐**. 지도에 원래 그려진 카카오 POI(상호 라벨)는 탭해도 안 잡히고, **줌/현재위치 재중심 버튼이 없다**.

---

## 1. 사용자가 짚은 두 문제 (최우선)

### 문제 1 · 지도에 보이는 POI를 탭해도 선택이 안 됨 ⚠️
"파크에비뉴B타워 오피스텔", "남원추어탕"처럼 지도에 이미 뜬 상호를 탭해도 아무 일이 없다. 검색을 거쳐야만 담을 수 있어 가장 부자연스럽다.

- **원인**: WebView→RN 이벤트가 `select`(우리가 그린 검색핀 클릭)·`longpress`뿐. 카카오가 타일에 구운 기본 POI 라벨은 DOM 요소가 아니라 클릭을 바인딩할 대상이 없다. (Kakao JS SDK는 구글맵의 POI/placeId click 같은 API를 공개 안 함 → 직접 잡기 불가.)
- **해결(권장)**: 지도 `click` 이벤트로 **클릭 좌표**를 받아, 그 좌표 **주변을 역으로 검색**해 사용자가 노린 상호를 추론한다.
  - WebView: `kakao.maps.event.addListener(map, 'click', …)` → 좌표를 `{type:'nearby', lat, lng}`로 RN에 올림.
  - 백엔드: `placeApi.nearby(lat,lng)` = `GET /api/places/nearby?lat&lng&radius=50` 추가. 카카오 로컬 `keyword.json`이 `x,y,radius,sort=distance`를 지원 → **카테고리 무관 최근접 상호**를 받는다. (SDK의 `categorySearch`는 FD6/CE7 등 그룹코드가 필요해 오피스텔·빌딩을 못 잡음 → 사용자 예시가 정확히 이 사각지대. 그래서 프록시 방식이 커버리지 최상.)
  - RN: `nearby` 수신 → 프록시 검색 → 결과 1개면 기존 장소 카드(`select` 흐름 재사용), 여러 개면 미니 후보 리스트. 최근접이 없으면 롱프레스와 동일하게 역지오코딩 주소 + 이름 직접입력으로 폴백.
  - 롱프레스 타이머(550ms)와 공존: 롱프레스 성공 직후의 `click`은 억제 플래그로 무시.
- **난이도**: 중 / **리스크**: 낮음(프록시·브릿지 아키텍처가 이미 있어 국소 확장).

### 문제 2 · 검색하면 화면이 너무 멀게(줌아웃) 나옴 ⚠️
검색 결과로 카메라 이동 시 과도하게 축소된다.

- **원인**: `renderResults`가 `map.setBounds(bounds)`를 **padding 없이** 호출, **최소/최대 level 클램프 없음**(1개일 때만 `setLevel(4)`). 키워드 검색이 좌표 힌트 없이 순수 텍스트라 동명 상호가 전국 단위로 섞여 bounds가 전국으로 벌어짐 → 극단 줌아웃.
- **해결**:
  1. (즉효) `setBounds(bounds, top, right, bottom, left)`로 **여백** 부여(하단 시트가 지도를 가리니 bottom 크게), **level 상한(예 7)·하한(3) 클램프**, 첫 결과에서 멀리(예 30km↑) 벗어난 결과는 bounds에서 제외.
  2. (근본) 검색 쿼리에 **현재위치(x,y,radius,sort=distance)**를 실어 "내 주변"이 먼저 오게 → 결과 분산 자체를 축소. RN에 이미 `me` 좌표 보유, 백엔드 프록시에 파라미터만 추가. 위치 권한 없으면 기존 키워드로 폴백.
- **난이도**: 낮음(옵션1) / 낮음~중(옵션2) · **리스크**: 낮음. (`setBounds` padding 시그니처는 고정 SDK 버전 문서로 확인.)
- 참고: 지도 탭 `KakaoMap.tsx`도 같은 카메라 패턴이라 같은 보정을 함께 적용하면 일관.

## 3. 이 앱(커플 회고형) 기준 판단

목적은 **탐색·길찾기가 아니라 "둘이 다녀온 곳을 빠르고 정확히 담기(회고)"**. 이 기준으로:

**Must-have** (지금 체감 빠짐)
- **지도 POI 직접 탭(문제1)** — 눈에 보이는 걸 못 고르는 가장 큰 이질감.
- **현재위치 재중심 버튼** — 지도 옮긴 뒤 "방금 그 자리"로 못 돌아옴. 다녀온 곳은 대개 지금 근처.
- **검색 위치가중(문제2 근본)** — 전국 동명 지점 난입 방지 = 정확도·줌 동시 개선. 저비용.
- **최근 담은 장소 원탭**(즐겨찾기 축소판) — 커플은 단골·데이트코스를 반복 → 재입력 0초.

**Nice-to-have**
- 줌 +/− 버튼(롱프레스 정밀 찍기 보조), 맵피커 내 결과 리스트(다건 구분), "이 지역 재검색", 즐겨찾기(별표).

**불필요/과함**
- 카테고리 필터·클러스터링·위성지도·영업정보/리뷰·근처 자동추천(이미 제외 확정). 회고형 목적과 안 맞거나 담기 핀 수가 적어 실익 없음.

## 4. 우선순위 (체감 큰 순서)

1. **검색 줌 보정(문제2 옵션1)** — 즉효·저리스크. `renderResults`에 padding+level 클램프.
2. **지도 POI 탭 → 담기(문제1)** — 백엔드 `nearby` 프록시 + WebView `click`→`nearby` 브릿지(`select` 흐름 재사용). 가장 큰 이질감 해소.
3. **검색 위치가중(문제2 근본, K)** — 프록시에 `x,y,radius` 추가. 정확도 근본 개선.
4. **현재위치 재중심 버튼(E)** — 지도 우하단 FAB, `setCenter(me)`만. 저비용·고체감.
5. **최근 담은 장소 원탭(I)** — 반복 방문 입력 0초.
6. (여유) 줌 버튼 · 맵피커 결과 리스트.

> 1·2·3은 모두 **백엔드 프록시 한 곳 + WebView 카메라/브릿지**의 국소 확장으로 묶여 있어 함께 처리하면 효율적. 확정되면 이 순서로 구현 착수.
