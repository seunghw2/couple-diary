# 27 · 오늘의 질문 — AI 운영 & 프롬프트북 (실전 가이드)

> "질문을 AI로 어떻게 다루나"의 실무 문서. 두 작업을 같은 규칙으로 처리한다.
> - **작업 A**: 기존 시드 질문을 새 스타일로 **리라이트**
> - **작업 B**: 주기적으로 신규 질문 **생성(배치 보강)**
> 근거: [25 AI 설계](25-daily-question-ai-plan.md)(톤·분류·파이프라인) + [26 스타일 분석](26-daily-question-style-analysis.md)(문형 4패턴). 백엔드 자리: `question/admin/`(AdminQuestionController·QuestionGenerator·QuestionFilter — 현재 생성기는 disabled 스텁, 키만 연결하면 동작).

---

## 0. 큰 그림 (파이프라인 한 장)
```
[프롬프트북(이 문서)] → LLM 호출(배치, JSON 강제)
   → 자동 필터(중복·형식·금칙·문형/톤 유효성·다양성)
   → 자동 통과 = active / 경계선 = 검수 큐(사람)
   → question_pool 적재(source='ai', batchId, promptVersion 기록)
   → 사용자 '별로예요' 신고 누적 → 자동 active=false (피드백 루프)
```
- LLM은 **아이디어 생성기**일 뿐, 최종 게이트는 **필터+사람+사용자 피드백**. LLM 출력을 무검증 적재 금지.

---

## 1. 어떤 AI를, 어떻게 부르나
- **모델**: 한국어 뉘앙스·반말·밈 이해가 좋은 상위 모델(예: Claude 계열). 리라이트/생성 모두 동일 모델.
- **호출 형태**: **배치**(한 번에 20~40개), **JSON 강제**(구조화 출력/함수호출), temperature **0.9~1.05**(다양성), 재시도 시 seed·순서 변주.
- **연동 지점(이미 스캐폴딩)**: `POST /api/admin/questions/generate`(헤더 `X-Admin-Token`). `QuestionGenerator` 인터페이스에 실제 LLM 구현을 끼우면 됨. 현재 `DisabledQuestionGenerator`라 호출 시 "미설정" 응답.
- **키/설정 필요**: LLM API 키(env), 모델명, 배치 크기, 월간/소진율 트리거. (실연동은 후속)

## 2. 공통 시스템 프롬프트 (A·B 공용, 복붙용)
```
너는 커플 교환일기 앱 '투데이'의 '오늘의 질문' 카피라이터다.
두 연인이 매일 하나의 질문에 각자 답하고 서로 공개하는 편지형 서비스다.

[스타일 — 반드시 지킬 것]
- 밋밋한 질문 대신 '작은 장면·놀이'를 던진다. 아래 4개 문형(form)을 섞는다:
  · direct  : 담백한 직접 질문 ("~한 순간은 언제야?")
  · scene   : 장면·상황 셋업(감탄사/의성어! + 가정 상황 + 질문) ("영화 1분 전! 그 사람이 안 온다면?")
  · blank   : 빈칸/지시대명사 채우기 ("난 네가 '이럴' 때 제일 설레더라 — 그 순간은?")
  · mission : 미션·행동 지시 ("지금 바로 그 사람 장점 3개를 적어보세요")
- 지칭은 당신/상대방/연인/그 사람/너를 문항마다 자연스럽게 섞는다(과한 존대 X).
- 가끔 답변형식을 짧게 지시한다(한 단어로 / 쭉 나열해 / 상대방 말투로 / 묘사해).
- 한국 커플 감성·밈·계절/시점 훅을 자연스럽게(깻잎, 커플유튜버, 시밀러룩, "오늘 저녁" 등). 억지 유행어 X.
- 한 문장(scene은 두 마디). 물음표 종결(mission은 청유형 "~보세요"). 이모지 남발 금지.

[light 기준 — 중요]
- 가벼워도 "부담 없이 답하지만 서로를 알게 되는 한 줄". 단순 사실확인("점심 뭐 먹었어?") 금지.

[금지]
- 과한 성적 표현, 정치·특정 종교 우열, 비난·불쾌 유도, 답이 곤란한 사생활 추궁, 정답 있는 상식/시사 퀴즈.
- 일기 본문·개인정보를 지어내 참조하지 말 것(우리는 객관 신호만 씀).

[분류 태그]
- tone: deep(관계·마음) | light(일상·취향·관찰) | fun(장난·상상)  ※ context(맥락/기념)는 별도 템플릿에서만.
- theme: 오늘/일상 · 취향/선호 · 서로 관찰하기 · 작은 가치관 · 나라는 사람 · 다정한 부탁 ·
         추억/함께한 시간 · 마음/애정표현 · 미래/약속 · 관계 돌아보기 · 상상/놀이
- depth: 1(가벼움)~3(깊음)

[출력]
- 오직 JSON 배열. 각 원소: {"text","form","theme","tone","depth"}
- 설명·머리말 없이 JSON만.
```

## 3. 작업 A — 기존 시드 리라이트
목표: 기존 `source='seed'` 질문 중 **밋밋한 direct 일변도**를 문형/생동감만 살려 다시 쓴다. **의미(주제)는 유지**, 표현·문형만 개선.

### 유저 프롬프트(배치 입력)
```
아래 기존 질문들을 위 스타일로 '리라이트'하라. 규칙:
- 원래 의도/주제(theme,tone)는 유지하되, 문장은 더 생동감 있게. 필요하면 form을 scene/blank/mission으로 바꿔라.
- 단순 사실확인·설문 느낌이면 확실히 놀이형으로 바꿔라.
- 한 문항당 후보 1개만. 너무 튀지 않게(과한 밈 금지).
- tone/theme은 원본과 동일하게 유지, form은 새로 판정해 표기.

입력(JSON): [{"id":12,"text":"요즘 제일 자주 듣는 노래는?","theme":"취향/선호","tone":"light"}, ...]
출력(JSON): [{"id":12,"text":"...","form":"scene|blank|mission|direct","theme":"취향/선호","tone":"light","depth":1}, ...]
```
### 처리
- 배치 20~40개. 출력의 `id`로 원본 매칭 → **새 text로 교체 저장**(원본 텍스트는 이력 보관 or 덮어쓰기). `form` 태그 채움.
- 리라이트도 **§5 자동 필터**를 통과해야 채택. 실패분은 원본 유지(리라이트 스킵).
- 한 번에 다 하지 말고 **문형이 direct뿐인 것 우선** 배치.

## 4. 작업 B — 주기 신규 생성 (배치 보강)
목표: 풀이 부족한 **theme·tone·form 조합을 겨냥해** 신규 생성. 무작정 많이가 아니라 **빈 칸 채우기(gap-driven)**.

### 생성 전 계산(서버)
- 현재 활성 풀의 tone 비율(목표 light 55% · deep 25% · fun 20%±)과 form 분포를 집계 → **부족한 축**을 프롬프트에 목표로 명시.
- 최근 N일 사용/기존 텍스트 목록(또는 대표 100개)을 **중복 회피용으로 프롬프트에 제공**.

### 유저 프롬프트
```
새 질문 30개를 생성하라. 목표 분포:
- form: scene 10, blank 6, mission 5, direct 9   (문형 다양성 필수)
- tone: light 16, fun 8, deep 6
- theme: '서로 관찰하기'·'상상/놀이'·'다정한 부탁'을 넉넉히(현재 부족).
아래 '기존 질문 목록'과 의미가 겹치지 않게(비슷한 발상 회피):
[기존 예시 100개 나열]
출력은 JSON 배열 {"text","form","theme","tone","depth"} 만.
```
### 맥락/마일스톤 템플릿은 별도
- `isTemplate=true, tone=context`는 일반 배치와 분리 생성. placeholder는 `{N}`·`{상대}`만. contextTrigger(anniversary/birthday/streak/season/firstletter/comeback)별로 2~3개씩. (스타일도 scene/mission 섞기)

## 5. 자동 필터 (LLM 출력 → 채택 전 게이트)
`QuestionFilter`에 구현(일부 완료). 순서대로 컷:
1. **형식**: 길이 8~60자, 물음표 종결(mission은 "~보세요/나열해" 허용), 한 문장(scene 2마디까지).
2. **금칙/민감**: 금지 가이드 위반 키워드·주제 차단.
3. **중복**: 기존 풀과 텍스트 정규화 후 완전/근접 중복 제거(가능하면 임베딩 코사인 유사도 임계).
4. **유효성**: tone/theme/form 값이 허용 집합인지.
5. **다양성 밸런스**: 이번 배치가 한 form/theme에 쏠리면 초과분 컷(목표 분포 대비).
- 통과 → `active=true`. 애매(길이 경계·유사도 경계) → **검수 큐**(active=false, 사람 확인 후 승격).

---

## 6. 사람 검수 & 사용자 피드백 루프
- **검수 큐**: 경계선 문항만 사람이 훑고 승인/반려(대량 아님). 반려는 폐기 or 재프롬프트.
- **사용자 신고**('이 질문 별로예요'): 서로 다른 커플 **3팀 이상**이면 자동 `active=false`(이미 구현). 신고 사유 태그가 쌓이면 프롬프트 개선 인풋으로.
- **좋은 질문 신호**: `usedCount` 대비 오픈율(둘 다 답한 비율) 높은 문항 = 승리 패턴 → few-shot 앵커로 재투입.

---

## 7. Few-shot 앵커 (프롬프트에 3~5개 주입 · 저작권 안전 자작)
> 참고 세트를 복제하지 말고 '패턴'만. 아래는 우리가 새로 쓴 예.
- (scene) "지하철 막차 3분 전! 그 사람이 안 온다면 너의 문자 한 줄은?"
- (scene) "무인도에 딱 하나만 가져간다면? 단, 그 사람 물건 중에서!"
- (blank) "우리 사이엔 '이 규칙' 하나는 꼭 지키자! 그게 뭐야?"
- (mission) "지금 바로 그 사람의 장점 세 개를 3초 안에 적어보세요."
- (direct+관찰) "요즘 그 사람에게서 새로 발견한 표정이 있어?"

---

## 8. 운영 파라미터 & 주기
- **트리거**: 활성 비템플릿 풀이 임계(예 120개) 미만 or 월 1회.
- **배치 크기**: 20~40(필터 통과율 60~80% 가정 → 실채택 15~30).
- **temperature 0.9~1.05**, 재시도 2회(중복·형식 실패 시). 다양성 위해 배치마다 목표 분포·few-shot 순서 변주.
- **버전 기록**: `batchId` + `promptVersion`을 각 문항에 저장(문제 배치 롤백·A/B 비교용).
- **비용 감각**: 배치가 작아 월 비용 미미. 무검증 대량 적재보다 **작게 자주 + 필터**가 안전.

---

## 9. 프라이버시·안전 원칙(재확인)
- 질문 생성/배정에 **일기 본문·기분 데이터 미사용**. 개인화는 기념일·생일·요일·연속일 등 **객관 신호만**.
- LLM에 커플 실데이터(답변 등) 주입 금지. 프롬프트엔 '기존 질문 텍스트'와 '목표 분포'만.

---

## 10. 실행 체크리스트
**리라이트(A)**
- [ ] direct 일변도 시드 배치 추출 → 시스템+A 프롬프트로 호출
- [ ] JSON 파싱 → 자동 필터 → 통과분 text/form 교체 저장
- [ ] 실패분 원본 유지, 배치 로그 기록

**주기 생성(B)**
- [ ] 부족 축(tone·form·theme) 계산 → 목표 분포 프롬프트 구성
- [ ] 기존 대표 100개 중복회피 컨텍스트 첨부 → 호출
- [ ] 필터 → active/검수 큐 분류 → 적재(batchId·promptVersion)
- [ ] 맥락/마일스톤 템플릿은 별도 배치

**공통**
- [ ] 관리자 엔드포인트에 실제 LLM 구현 연결(키·모델) — 현재 disabled 스텁
- [ ] 사용자 신고 임계·검수 큐 모니터링
