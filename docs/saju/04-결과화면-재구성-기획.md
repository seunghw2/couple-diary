> 사용자 피드백("예쁘고 읽기 편한데 아직 너무 무난한 결과지")에 대한 재구성 설계.

## 1. 먼저 고칠 버그 — 막대 색이 사실상 안 먹는다

두 분석이 독립적으로 같은 결론에 도달. **이건 개선이 아니라 버그다.**

- `couple.tsx:15~20` `gradeColor()`는 `grade >= 4`, `grade >= 3` 분기를 검사.
- 백엔드 `SajuCompatibility.java:102` `grade()`는 **0/1/2만** 반환(`s>=67?2:s>=34?1:0`).
- 결과: `c.primary`/`c.coralSoft` 분기는 **죽은 코드**. 실제로는 grade 2(노랑)·grade 0·1(갈색) **2색만** 나옴.
- 사용자가 본 "막대가 대부분 갈색 + 대화만 노랑"이 정확히 이 증상. 대화(100→grade 2)만 노랑, 나머지(47·55·63·66)는 전부 갈색이라 **점수 차이가 색에 안 보임.**

→ 색을 grade가 아니라 **score 직접 구간**으로 매핑(3장에서 상세). 이 하나로 "왜 47인데 다 같은 색?" 의문이 사라짐.

## 2. 점수 신뢰감 — 종합 83% vs 항목평균 66점

### 2-1. 원인 (버그 아님, 의도된 설계)

항목 점수와 종합은 **다른 수식**이다.
- 항목 = 0~100 원점수(막대에 그대로).
- 종합 = 가중합 → (raw−30)/60 정규화 → **`60 + norm*39`로 60~99 대역 리매핑**.

사용자 사례 대입: raw 65.65 → norm 0.594 → **83%** (화면값과 정확히 일치).
하한이 60이라 raw가 30만 넘으면 무조건 60%+. 항목의 낮은 값(47)이 종합엔 거의 안 비쳐 "항목엔 47인데 종합 83?"이 됨. 스펙(§3-1)의 "커플 앱은 높게 나와야 기분 좋다"는 **의도된 인플레이션**.

### 2-2. 해결: ⓘ 설명 추가 (산식 유지) — 사용자 선호와 일치

산식을 낮추면 대부분 커플 percent가 40대로 내려가 "우리 별로네" 역효과 + 배지/총평 구간 재튜닝 → 리스크 큼. 대신 **투명성**으로 신뢰 회복. 프론트만 수정.

- 종합 percent 옆 ⓘ → 툴팁:
  > **종합 점수는 어떻게 나오나요?**
  > 오행의 상생·상극, 일간의 합·충 등 12가지 관계를 종합해 계산해요. 아래 항목 점수는 '첫끌림·대화·애정·안정감·성장'을 따로 본 값이라, 단순히 평균한 숫자와는 달라요. 그래서 항목을 더해 나눈 값과 종합 점수가 일치하지 않는 게 정상이에요. 😊
- 항목 헤더 옆 짧은 ⓘ도 동일 취지 1줄.
- (선택) 문안을 `CoupleResult.scoreExplain` 필드로 서버 관리.

### 2-3. 점수 ↔ 설명 온도 매칭

문제: 애정 47 → grade 1(중) → `AFFECTION_T[1]`이 "표현 담백해도 마음은 깊은…"으로 **너무 긍정**. 온도 불일치의 원인.

- grade 컷 조정: `>=70 : >=45 : else` (현재 67/34).
- 등급별 톤 규칙 재정의:

| grade | 톤 | 반드시 포함 | 금지 |
|---|---|---|---|
| 상 | "이미 잘 통해요" | 강점 확언 + 유지 격려 | 조건부 최소화 |
| 중 | "좋은데 살짝 다듬으면 더" | 긍정 1문장 + **보완 행동 1개** | 칭찬만 |
| 하 | "지금은 다른 편, 채워갈 여지" | **차이 방향 명시** + 성장 프레임 + CTA | "나쁨/이별" |

- 애정 중(1) 재작성 예:
  > 마음의 크기보다 표현하는 방식이 조금 다른 두 분이에요. 한 사람은 챙기는 행동으로, 다른 사람은 말과 반응으로 사랑을 확인하는 편이에요. '알겠지' 하고 넘기기보다 한 번 더 말로 표현하면, 애정이 훨씬 또렷하게 전해져요.

## 3. 색·라벨 체계 (score 직접 구간)

`gradeColor(grade)` → `scoreColor(score)` / `scoreLabel(score)`로 교체(프론트만, 백엔드 무변경):

| 점수 | 라벨 | 색 | 의미 |
|---|---|---|---|
| 80~100 | 찰떡 | 골드 `#E8B96A` | 최상 |
| 60~79 | 안정적 | 브라운 `#B39685` | 무난·성숙 |
| 0~59 | 맞춰가기 | 로즈 `#D98C9A` (연한 장미) | 성장 여지 |

- 강한 빨강(`danger #E5654B`)은 **금지** — 불안 유발. 로즈로 순화.
- 라벨은 막대 옆 pill + 히어로 percent 옆 배지로 이중 노출(색+텍스트).
- 히어로 percent 색도 현재 항상 코럴 단색 → 구간색 적용해 최상단에서 밴드 즉시 전달.
- `theme.ts`에 `rose` 토큰 추가(하드코딩 지양).

## 4. 콘텐츠 재미·구체성 — 핵심 재설계

### 4-1. 근본 진단: 계산은 풍부한데 문장은 등급만 본다

`SajuCompatibility.cat()`은 문장을 **(grade 0/1/2 × seed)** 로만 고른다. 그런데 `compute()`는 이미 아래를 **다 계산해놓고 버린다**:
`chDay`(일지충=갈등 대처 다름), `sangbo`(상보=역할 분담), `combined[1]`(화 총량=표현형 vs 행동형), `yinYangComplementDay`(음양차=속도차), `distSim`(사고 결), `ganChung`(정면 부딪힘), `control`(과열 제어) 등.

→ **재설계 뼈대: 이미 계산되는 관계 feature를 문장 선택의 1차 키로 승격**(등급은 톤 조절용 2차 키로).

### 4-2. "형용사 → 장면" 원칙

범용 문장은 상태 형용사("편안한/다정한/단단한"). 구체 문장은 **비대칭 + 트리거 상황 + 행동 동사** 3요소 중 2개 이상 포함.
- 나쁨: "서로를 다독여주는 사이예요."
- 좋음: "다툰 직후, 한 분은 혼자 정리할 시간이 필요하고 다른 한 분은 바로 풀고 싶어 해요. 타이밍만 맞추면 회복이 빨라져요."

### 4-3. feature → 행동문장 매핑 (핵심 산출물)

각 feature를 5개 행동 도메인(싸울 때/데이트/표현/속도/시간축)에 매핑. 비대칭은 방향 규칙까지 결정론으로 확정.

| feature (조건) | 도메인 | 행동문장 | 비대칭 방향 |
|---|---|---|---|
| `chDay` 일지충 | 싸울 때 | "다툰 직후 한 분은 혼자 정리할 시간이, 다른 한 분은 바로 풀고 싶어 하는 조합. 타이밍만 맞추면 회복이 빨라요." | 일지 음=정리형 / 양=바로푸는쪽 |
| `ganChung` 천간충 | 의견충돌 | "부딪힐 땐 둘 다 물러서지 않아 스파크가 튀지만 그만큼 솔직한 사이. '누가 맞나'보다 '뭘 원하나'로 물으면 금방 풀려요." | 대칭 |
| `yinYangComplementDay` 음양차 | 속도차 | "한 분은 정해지면 바로 움직이고, 다른 분은 살펴본 뒤 확신이 서요. 서두름과 신중함을 서로 채워줘요." | 일간 양=바로움직임 |
| `sangbo≥0.5` 상보 | 역할분담 | "역할이 겹치지 않아 계획은 한 분이, 분위기는 다른 분이 맡는 팀플레이형." | 오행 많은 쪽=채우는쪽 |
| `combined[1]<2` 화부족 | 표현 | "마음은 깊은데 '좋아해'를 말보다 행동으로 보여주는 조합. 하루 한 번 말로 확인하면 더 안심돼요." | 대칭 |
| `combined[1]≥3` 화많음 | 표현 | "좋고 싫음이 얼굴·말에 바로 나와 감정 숨기기 어려운 사이. 욱한 말도 빨리 나오니 한 템포만 쉬면 완벽." | 대칭 |
| `chMon` 월지충 | 일상리듬 | "생활 리듬이 살짝 어긋나는 편(아침형·밤형). 같이 있는 시간대를 정해두면 편해져요." | 월지 음/양으로 배정 |
| `distSim<0.35` 분포상이 | 대화결 | "말은 통하는데 서운함은 늦게 꺼내는 조합. '나는 지금'으로 바로 꺼내면 안 쌓여요." | 대칭 |
| `distSim≥0.6` 분포유사 | 대화결 | "'나도 방금 그 생각!'이 자주 나와요. 다만 비슷한 데서 같이 놓치니 가끔 다른 시각을 꺼내보면 좋아요." | 대칭 |
| `control` 과다제어 | 브레이크 | "한 분이 과열될 때 다른 분이 자연스레 브레이크가 되어주는 조합. 균형을 잡아주는 거예요." | 억제되는 오행 쪽=과열 |
| `yukDay`/`samDay` 일지합 | 정서밀착 | "둘만 있을 때 유독 편안해지는 조합. 밖에서보다 단둘이 있을 때 진짜 모습이 나와요." | 대칭 |

비대칭 방향은 `STEM_YANG[]`·`elementCount` 대소·일지 홀짝으로 결정 — **새 계산 불필요, 결정론 유지**. "한 분/다른 분"을 실제 닉네임으로 바꾸려면 문장에 이름 슬롯 도입.

### 4-4. 조합 방식: "base(등급) + signature(feature)"

카테고리 코멘트 = 등급별 base 문장(톤) + 가장 강한 feature의 행동문장(구체성).
- 카테고리마다 우선순위로 첫 매칭 feature 하나 선택.
- feature 미매칭 시 기존 base로 폴백 → **한 카테고리씩 켜며 안전하게 점진 도입**.

### 4-5. 점수-스토리 총평 (최고 × 최저 엮기)

현재 총평은 percent 구간으로만 갈려 "대화 92·애정 61"과 "대화 61·애정 92"가 같은 총평을 받음. → **최고·최저 카테고리를 한 문장으로 엮기.**

구조: `[최고 칭찬 절] + [전환] + [최저 '차이' 절(성장)] + [잇는 메시지]`

예시:
1. 대화高·애정低: "대화는 놀랄 만큼 잘 통하는데 애정 표현 방식은 서로 조금 달라요(한 분은 말로, 한 분은 행동으로). 말하지 않아도 아는 사이에서, 말해줘서 더 안심되는 사이로 가면 오래갑니다."
2. 첫끌림高·안정低: "처음부터 자석처럼 끌린 두 분. 생활 리듬은 아직 맞춰가는 중이라, 설렘이 식기 전에 둘만의 루틴을 하나 만들면 두근거림이 안정감으로 자리 잡아요."
3. 안정高·첫끌림低: "불꽃보다 온기로 시작한 사이. 곁에 둘수록 편안해지는 조합이라 6개월 뒤에 훨씬 단단해져요. 지금이 제일 예쁜 초반이에요."
4. 성장高·대화低: "역할이 겹치지 않아 한 분이 채우면 다른 분이 메우는 팀플레이형. 서운함은 늦게 꺼내는 편이라 '나는 지금' 한마디면 이 팀워크가 더 오래가요."
5. 격차 작음: "어느 하나 튀지 않고 고르게 좋은 균형형 커플. 가끔 계획 없는 데이트로 새로움을 더하면 완벽해요."

### 4-6. 대표 한줄 (히어로용)

"가장 희귀·강한 signal 하나"를 뽑아 한 줄. 예:
- 대화高·애정低: "말은 잘 통하는데, 표현은 더 해줘야 안심되는 사이."
- chDay: "싸운 뒤 푸는 속도만 맞추면 회복이 빠른 커플."
- sangbo高: "한 명이 비면 한 명이 채우는, 손발 맞는 조합."
- 첫끌림低·안정高: "천천히, 그래서 오래가는 슬로우 케미."

### 4-7. 관계 꿀팁 → 약점 카테고리 연결 처방

현재 `pickTips()`는 생일 시드 무작위 3개(진단과 무관). → **[최저 처방] + [2번째 낮은 처방] + [최고 유지 팁]**, 각 "왜(점수 근거)+뭘(행동)" 구조.

- 애정 낮음: "하루 한 번 마음을 말로 확인해 보세요. 행동으론 잘 챙기는 두 분이라, 표현만 더하면 놓칠 일이 없어요."
- 대화 낮음: "서운할 땐 '너는 왜'보다 '나는 지금 ~해'로 시작해 보세요. 말의 온도가 다른 두 분에겐 이 한 끗이 커요."
- 안정 낮음: "다툰 뒤 '몇 시간 뒤 다시 얘기하자'를 미리 약속해 두세요. 푸는 속도가 다른 조합엔 타이밍 룰이 명약이에요."
- 첫끌림 낮음: "가끔 계획 없는 즉흥 데이트를 넣어보세요. 익숙함에 설렘 한 스푼이 필요한 조합이에요."

### 4-8. 톤 안전장치

기존 원칙(부정어 0, 상극=밀당 케미, 저점=성장형) 유지 + 추가:
- **'차이'를 지우지 말 것**: 순화하되 "한 분은 A, 다른 분은 B" 비대칭을 살려야 구체적으로 보임. "둘 다 좋은 사람"식 회귀 금지.
- 부정 트리거("싸운 직후")는 반드시 **회복/해법 절과 한 세트**.
- 최저 카테고리 문장 옆에 최고 카테고리 병기(총평 구조가 강제).
- 한자·전문용어(일지충 등) 비노출 — 행동 서술로만.
- DISCLAIMER 배너 유지.

## 5. 화면 구조 재구성 — 핵심 먼저, 상세는 접기

### 5-1. 진단

현재 순서: 히어로 → **항목별(5×3~4문장)** → 총평 → 꿀팁 → 배지 → 공유.
- 최대 병목: 항목별 카드가 전체 텍스트의 **~65%**. 히어로 직후 바로 긴 글 벽.
- 총평 ↔ 항목 코멘트 톤 중복("채워가는/따뜻한/성장형" 반복).
- 5개 카테고리가 동일 비중 나열 → "가장 잘 맞는/다른 곳" 판단 요약 없음(`strongestKey`는 DTO에 있으나 화면에서 미사용).
- 배지가 문장형인데 chip에 넣어 wrap 깨짐.

### 5-2. 재구성 (커플, 7단계)

| # | 블록 | 상태 |
|---|---|---|
| ① 대표 결과 | 이모지쌍 + 이름 + **유형명(큰 바다형 × 넝쿨꽃형)** + % + 대표 한줄 | 항상 |
| ② 우리 관계 한눈에 | 가장 잘 맞는 곳 · 가장 다른 곳 · 관계 키워드 (pill) | 항상, 신규 |
| ③ 딱 3줄 해석 | 3문장 요약 | 항상, 신규 |
| ④ 항목별 궁합 | 이름·점수·막대 + 코멘트 **2줄 프리뷰 + '자세히 보기'** | 아코디언, 기본 접힘 |
| ⑤ 관계 꿀팁 | 처방 3개 | 항상 |
| ⑥ 사주 근거 보기 | 배지(리스트) + 오행 상보 + disclaimer | 기본 접힘 |
| ⑦ 공유 | 이미지 카드 (하단 고정 버튼) | 버튼 |

개인 화면(`me.tsx`)도 동일 원리: desc 다문단 → 2줄 프리뷰+더보기, 원국·오행·생시는 "사주 자세히"(접힘)로 강등.

### 5-3. 텍스트 35~40% 감축

정보 손실 없이(삭제 아닌 접기):
- 항목 코멘트 2줄 프리뷰화 (최대 ~25%p)
- 총평 → ③ 3줄 해석으로 압축, 원문은 ⑥으로 (~8%p)
- 배지 문장 → ⑥ 리스트로 이동 (~3%p)

### 5-4. 한눈에 보기 산출

- 가장 잘 맞는 곳: `strongestKey`(이미 존재) 또는 `categories` max.
- 가장 다른 곳: `categories` min (프론트 계산 가능, 명시성 위해 `weakestKey` 권장).
- 키워드: strongestKey/badges 매핑 or 신규 `keywords` 필드.

### 5-5. 아코디언 구현 (새 의존성 0)

- RN 순정 `LayoutAnimation.configureNext(...easeInEaseOut)` + `useState`.
- 프리뷰: `comment.split(/(?<=[.!?…])\s/).slice(0,2).join(' ')` (문장 단위 2문장).
- 공용 `<Accordion>` 컴포넌트 신설(항목별 + 근거보기 + me desc 공용). 코드베이스에 기존 아코디언 없음.

## 6. 공유·바이럴

### 6-1. 공유 버튼 (즉시, 프론트만)

- `couple.tsx:188` `variant="soft"`(연배경+연글자=비활성처럼) → **`variant="primary"`**(진한 코럴+흰 글자).
- 문구 "궁합 공유하기" → **"우리 궁합 카드 공유하기"**.
- ScrollView 안 → **하단 고정 sticky CTA**(상단 그림자).

### 6-2. 공유 이미지 카드 (EAS 빌드 후속)

방안 비교 결론: **(a) RN 뷰 캡처** 권장(테마·이모지 재사용, 서버 렌더 리스크 없음).
- 신규 의존성: `react-native-view-shot` + `expo-sharing` (둘 다 네이티브 → **EAS 재빌드 필요**, Expo Go 불가). `expo-media-library`·`image-manipulator`·`svg`는 이미 설치됨.
- 카드 레이아웃: 워드마크 / 이모지쌍 / 이름×이름 / **유형명** / 대형 % + 구간 라벨 / 캐치프레이즈 한 줄(relComment, 총평 전문 아님) / 하단 앱 워터마크.
- 데이터 갭: `SajuCouple`에 유형명 없음 → `meTypeKo`/`partnerTypeKo` DTO 추가 필요(개인 `dayMasterKo`를 궁합에도 전달).

### 6-3. 이모지 통일

난립 원천은 **백엔드 문구 끝 이모지**(🍀💌💬🌱🧩🧱⚖️💪🔄🌿🚀🌈👣💧☯️🎇…) — "AI가 문단마다 붙인 느낌".
- 원칙 1: 본문 문구에서 이모지 제거(텍스트만, 재미는 색·배지·아이콘이 담당).
- 원칙 2: 카테고리당 대표 아이콘 1개 고정 — 첫끌림 ✨ / 대화 💬 / 애정 ♥ / 안정감 🏠 / 성장 🌱.
- 원칙 3: 섹션 헤더 장식 이모지 → Ionicons로 통일.
- 예외: 배지는 "특별함" 파는 요소라 이모지 유지(카테고리 세트와 비중복).

## 7. 백엔드 DTO 확장 (`CoupleResult`)

```java
String strongestKey,      // (기존)
String weakestKey,        // 신규: 가장 다른 곳
List<String> keywords,    // 신규: 관계 키워드 2~3개
String meTypeName,        // 신규: meDm.name() — 서비스가 이미 계산 중(거의 공짜)
String partnerTypeName,   // 신규: pDm.name()
List<String> summaryLines // 신규: 3줄 해석(선택, 없으면 프론트 조합)
```
- `meTypeName`/`partnerTypeName`은 서비스가 이미 갖고 있어 사실상 무비용(히어로·공유 카드 "유형 × 유형").
- 개인 `PersonalResult`는 `oneLine`/`keywords`/`strengths`/`growthPoints` 이미 있어 **DTO 변경 불필요**.

## 8. 구현 로드맵 (리스크 최소 순)

### 1단계 — 프론트 단독, 무빌드·무배포 (불만 대부분 해소)
- [ ] `scoreColor`/`scoreLabel` 구간 매핑 (색 버그 수정 + 라벨 pill)
- [ ] 히어로 percent에 구간색 적용
- [ ] 공유 버튼 `variant="primary"` + 문구 + 하단 고정
- [ ] 종합/항목 ⓘ 툴팁
- [ ] 화면 구조: ② 한눈에 보기 + ③ 3줄 해석 + ④ 항목 아코디언(2줄 프리뷰) + ⑥ 근거 접기 (`<Accordion>` 신설, categories로 최고/최저 프론트 산출)
- [ ] 프론트 이모지 정리(헤더·하트 통일)

### 2단계 — 백엔드 배포 1회
- [ ] `cat()`에 feature 묶음 전달 → base+signature 문장(폴백 유지, 카테고리별 점진)
- [ ] 총평 최고×최저 엮기(`worst` 산출) + 대표 한줄
- [ ] `pickTips()` → 약점 카테고리 연결 처방
- [ ] 문구 끝 이모지 제거
- [ ] grade 컷 조정(70/45) + 하·중 문구 톤 재작성
- [ ] DTO: `weakestKey`/`keywords`/`meTypeName`/`partnerTypeName`

### 3단계 — EAS 빌드 후속 (바이럴 핵심)
- [ ] `react-native-view-shot` + `expo-sharing` → 공유 이미지 카드
- [ ] 카드 유형명 위해 DTO 유형명 필드 소비

## 9. 가장 효과 큰 한 방

사용자가 직접 꼽음 — 첫 화면 아래 한 줄:
> "대화는 100점이지만 애정 표현은 서로 다른 두 분이에요. 마음을 알아주는 것에서 멈추지 말고 한 번 더 말로 표현하면, 오래갈수록 더 단단해지는 궁합입니다."

= **4-5 점수-스토리 총평 + 4-6 대표 한줄**의 결합. 숫자들이 하나의 이야기로 묶이는 순간 콘텐츠가 그럴듯해진다. 1단계(③ 3줄 해석)만으로도 이 효과의 상당 부분을 무빌드로 달성 가능.
