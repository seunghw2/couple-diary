# 03 · 제품·UX 설계 (월드컵 패턴 미러링)

> 전체 원칙: **월드컵을 그대로 미러링**하되, 콘텐츠가 "토너먼트 진행"이 아니라 "생년월일→고정 템플릿 계산 결과"라는 점만 다르다. 진입/배지/알림/커플비교/딥링크는 100% 동일 구조 재사용.

---

## 파트 1. 월드컵 통합 패턴 (코드 조사)

### 프론트 구조
- **진입점 — 설정 탭 "바로가기" 그룹** `frontend/app/(tabs)/settings.tsx:52-74`. `SettingsRow` "월드컵 게임" 행 + `trophy-outline` + 우측 배지 + `router.push('/worldcup')`. 배지는 `worldcupApi.unseen()`을 `useFocusEffect`로 조회(`:18-23`), `wcUnseen>0`일 때만 원형 카운트(`:64-70`). partner 없으면 조회 skip.
- **목록** `frontend/app/worldcup/index.tsx` — `useFocusEffect`에서 `worldcupApi.list()` + `worldcupApi.markSeen()`(`:25-31`). **"목록 열면 배지 사라짐"** 규칙. 카드 = 이모지+제목+`statusLabel`(내/상대 완주)+N강 칩.
- **상세/결과** `frontend/app/worldcup/[key]/index.tsx` — 파라미터 `key, justWon, winnerLabel/Emoji, compare`(`:19-29`). `justWon==='1'` 또는 `compare==='1'`이면 결과 비교 자동 펼침. 완주 직후 우승 배너(`winBanner :76-81`). **커플 비교 컴포넌트 `JourneyCompare`(`:147-196`)**: 좌(나)/우(상대) 2열, 티어별 칩, 상단 "취향 일치율 %". ← **커플 비교 결과 화면의 원형.**
- **플레이** `frontend/app/worldcup/[key]/play.tsx` — 완주 시 `worldcupApi.saveResult(...)` 후 `router.replace`로 `justWon:'1'` 전달(`:46-63`). 저장 실패해도 결과는 보여줌.
- **API** `frontend/lib/api.ts:580-590` — `worldcupApi.{list,detail,saveResult,records,unseen,markSeen}`, 타입 `:540-578`.
- **알림 딥링크**(동일 규칙 2곳): 인앱 `notifications.tsx:52-55`, 푸시 `_layout.tsx:65-67` — `WORLDCUP_*`면 `refKey`를 key로 `{key,compare:'1'}` push. 아이콘 `notifications.tsx:28-29`.

### 백엔드 구조
- **Controller** `/api/worldcups`: list, `/unseen`, `/seen`, `/{key}`, `/{key}/result`, `/{key}/records`. 리터럴 경로 우선(`:26`).
- **Service** `WorldcupService.java`: `saveResult(:65-93)`가 저장 후 `notificationService.onWorldcupCompleted(me,partner,...,key,partnerDone)` — partnerDone이면 "비교 가능" 승격. `records(:107-139)`는 둘 다 완주 시만 `CompareView`.
- **카탈로그** `WorldcupCatalog.java` — 콘텐츠를 코드 상수로. **사주는 콘텐츠가 계산 생성이라 카탈로그가 아닌 계산 서비스로.**
- **엔티티** `WorldcupResult.java`: author, couple, key, winnerId, stages(문자열), createdAt.
- **알림** `NotificationService.java:225-265` + `NotificationType.java:16-17`: `WORLDCUP_TYPES`, `onWorldcupCompleted`(refKey 딥링크), 배지 = 미읽음 카운트, 목록 열람 시 read.

### 유저/커플 데이터
- **`User.birthday` 이미 존재**(`User.java:48`, LocalDate nullable). 프론트 `UserSummary/PartnerSummary.birthday`(`api.ts:122,131`, YYYY-MM-DD). **상대 생일도 프론트 접근 가능** → 궁합 즉시 활용.
- 생일 입력 UI `account.tsx:188-196`(DatePickerSheet). **단 "생시" 개념은 어디에도 없음** → 신규 필요.
- **`Couple`**: user1/user2, `partnerOf/memberOf` 헬퍼, `requireCouple(userId)` 가드.

## 파트 2. 사주 제품/UX 설계

### 2-1. 진입점 & 네이밍
설정 "바로가기"의 **월드컵 게임 바로 아래** 새 행(`settings.tsx:60-73` 복제):
| 항목 | 값 |
|---|---|
| 아이콘 | `sparkles-outline`(또는 `moon-outline`) |
| 라벨 | **"우리 사주 궁합"** (대안: "오늘의 사주") |
| 배지 | `sajuApi.unseen()` (월드컵과 동일 매커니즘) |
| 라우팅 | `router.push('/saju')` |

허브 서브텍스트: "재미로 보는 사주 · 우리 둘의 궁합 🔮"

### 2-2. 사용자 플로우
**허브 `/saju`**(월드컵 index 미러): 진입 시 `markSeen`. 카드 2장 — **① 내 사주** / **② 우리 궁합**. 상태 라벨: 생일 미입력이면 "생일 먼저 등록", 상대 생일 없으면 "상대가 생일을 아직 안 넣었어요".

**(a) 개인 사주 `/saju/me`**: `User.birthday` 있으니 바로 계산. **생시 선택 입력**(결과 상단 접이식 카드) — 12지시 세그먼트 + **"모름"(기본)**. 모름이어도 일간 기반 결과는 항상 나옴. 저장처: `User.birthTime`(nullable) 신규 컬럼.

**(b) 커플 궁합 `/saju/couple`**: 내+상대 생일 필요(상대는 `PartnerSummary.birthday`로 접근). **상대 생일 미입력 시**: 안내 카드 + **"상대에게 생일 입력 요청"** 버튼(`NotificationService.poke` 패턴, 새 타입 `SAJU_BIRTHDAY_REQUEST`). 생시는 각자 선택, 한쪽만 알아도 계산 진행.

### 2-3. 결과 화면
**개인 사주** — 오행 분포(가로 막대 5개, 순수 View width%로 구현·의존성 X, 🌳🔥⛰️⚙️💧) + 일간 캐릭터(큰 이모지 + "당신은 큰 나무 같은 사람 🌳(갑목)", winBanner 스타일) + 한줄 성격(고정 템플릿) + (선택)오늘의 기운 + 공유.

**커플 궁합** — **`JourneyCompare` 미러**: 상단 궁합 % 대형 숫자(월드컵 "일치율 87%" 위치) + 좌(나)/우(상대) 2열 일간 캐릭터 마주보기 + 카테고리별 점수 가로 막대(성격·가치관·애정·대화 케미) + 조합별 따뜻한 코멘트 + 공유.

### 2-4. 재방문 & 알림/배지
- **본체(오행·일간·궁합%)는 고정**(생일 불변). **"오늘의 운세"만 `날짜+일간` 시드로 매일 갱신** → 데일리 훅.
- 새 `NotificationType`: `SAJU_BIRTHDAY_REQUEST`("상대가 궁합 보고 싶어해요, 생일 넣어줄래요?"), (선택)`SAJU_COMPATIBILITY_READY`(상대가 생일 채워 궁합 볼 수 있게 됨 — 월드컵 COMPARABLE과 동형).
- **배지**: 설정 사주 행 = 사주 알림 미확인 수, `/saju` 열면 markSeen. `NotificationService`에 `SAJU_TYPES`+`countUnreadSaju`+`markSajuSeen` 추가(월드컵 복제).
- **딥링크**: `refKey`에 `"me"`/`"couple"` → notifications·_layout의 WORLDCUP 분기 옆 SAJU 분기 추가.
- 데일리 로컬 알림은 MVP 제외(과알림 위험), 인앱 배지 우선.
